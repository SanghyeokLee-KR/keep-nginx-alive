# 제출용 Write-up — 5조 "NginX를 살려라"

> 교육용·방어 전용. 권한이 부여된 경기 박스 안에서 우리 팀 nginx 가용성을 방어한 기록.

## 핵심 성과

- **채점 다운 0회** — 금~일 경기 내내 `#1` 백엔드는 채점 기준 다운 0회.
- **복구 ~2초 (실측)** — `nginx kill` 시 Teams 알림이 다운→복구 2초를 기록(④ 관측). 채점 임계치(연속 3회·분 단위) 대비 큰 여유.
- **1·2·3차 공격 전부 방어 + 관측** — 자동복구 무력화·80 포트 하이재킹·`systemctl mask`를 각각 다중 복구·NLB 우회·자력 unmask로 막아냈고, auditd로 전 과정을 캡처했다.

## ① 방어 전략 · 예상 공격 · 구현 방법

**전략.** 규칙상 공격자를 차단·권한제한할 수 없다. 그래서 "쫓아내기"가 아니라 "탐지·복구를 자동화하고, 끝내 오염되면 채점망에서 빠지기"로 잡았다. 다중 자동복구 + 보호 모니터링 서비스 + 자가격리 페일오버.

**예상한 공격 → 대응 설계:**

| 예상 공격 | 대응 |
|---|---|
| nginx kill/stop 반복 | systemd `Restart=always`(2s) + watchdog A·B(5s) + cron(1m) — 5중 복구 |
| index.html 변조 | `chattr +i`(우발 변조 방지·공격 지연) + watchdog 본문검증 복원 |
| 80 포트 하이재킹(apache 등) | watchdog 점유자 제거 + 헬스체크 자가격리 → NLB 우회 |
| `systemctl mask` | `unmask-guard` 자력 복원 |
| 방어 서비스 탐지·제거 | 서비스명 추상화(`Description`·dotfile) + A·B 상호 복구 + 독립 복구 경로 |

**구현.** [① 아키텍처](01-architecture.md) · [② 다층 방어](02-defense-in-depth.md) · [③ 페일오버](03-healthcheck-failover.md) · [④ 관측·감시](04-monitoring.md) · [⑤ 타임라인](05-attack-timeline.md) (다이어그램 10장 + [스크립트](../scripts/README.md))

**결과.** 경기 내내 우리 `#1`은 채점 다운 0회였다. 3차에 `systemctl mask nginx`까지 당했지만, `unmask-guard`를 미리 깔아둔 덕에 설계대로 watchdog가 자력 `unmask` 후 복구했다.

가장 공들인 부분은 보호 모니터링 서비스 설계다. 같은 스크립트를 일반적인 시스템 서비스명(`System Resource Monitor` 등)과 dotfile로 추상화해 공격자의 단순 탐색 비용을 늘렸다. 다만 이건 보안 경계가 아니라 탐지 지연 수단일 뿐이고, `systemctl`로는 그대로 보인다는 한계도 안다.

> **증거 — 2차 apache2 80 포트 하이재킹 (어태커 명령 실시간 캡처).** `systemctl stop/disable nginx` → `fuser -k 80/tcp` → `apt-get install apache2` → `systemctl start apache2`로 80 포트를 탈취하는 명령이 그대로 기록됐다. (어태커 IP 마스킹)
>
> ![2차 공격 명령 로그](../screenshots/attack-2-apache-port-hijack.png)

> **증거 — 다운 → 복구 2초 (Teams 알림 실측).** nginx 다운 감지 후 복구 완료까지 2초가 그대로 기록됐다. 사람이 화면을 못 보는 새벽에도 상태 변화가 즉시 푸시됐다.

## ② 배운 점

- **프로세스가 떠 있는 것과 서비스가 살아 있는 것은 다르다.** 응답 코드 200만으로는 부족하고 본문(`h1`)까지 확인해야 진짜 가용성을 확보할 수 있다. 이것이 watchdog를 본문 검증 방식으로 만든 이유다.
- **방어 코드도 결국 같은 서버 자원을 먹는다.** 감시·복구 데몬을 너무 자주, 무겁게 돌리면 CPU·메모리를 깎아 오히려 가용성을 떨어뜨린다. 데몬 주기와 로직을 가볍게 잡는 게 방어 로직만큼 중요했다.
- **가용성은 결국 비용이다.** 백엔드를 늘리고 인스턴스를 키우고 트래픽 방어를 붙이는 건 전부 돈이다. "어디까지 방어할지"가 기술 문제이자 예산 문제라는 걸 체감했다.
- **작은 설정 한 줄이 관측 전체를 좌우한다.** auditd `auid>=1000` 필터 한 줄이 명령 로그 품질을 살리고 죽였다.
- **잠그는 게 늘 안전한 건 아니다.** `index.html`은 `chattr +i`로 잠갔지만 헬스체크 `ok` 파일은 일부러 잠그지 않았다. 잠갔다면 오염된 페이지가 채점봇에 그대로 노출됐을 것이다. 실패 모드까지 설계해야 한다.

## ③ 아쉬운 점

- 모든 트래픽이 **단일 서비스 포트(`:80`)에 집중**돼 있다. 대회 규칙이 가용성 공격(DDoS 등)을 명시적으로 제외했기에 이번엔 대비하지 않았지만, 실제 운영이라면 볼륨형·L7 공격에 NLB 뒤 백엔드 자원이 고갈될 수 있는 한계다. rate limiting·연결 수 제한·WAF 같은 트래픽 레벨 방어가 없었다.
- 경기 중엔 방어 스택을 **수작업으로 구성**해 재현이 어려웠다. 이 아쉬움은 경기 후 ansible playbook으로 코드화해 일부 해소했지만([`ansible/`](../ansible/)), 인스턴스 교체형(ASG)·Multi-AZ까지는 아직 미구현이다.
- **`#2`·`#3`의 독립 복구 검증이 부족**했다. 주로 `#1` 우회에 의존했고, 클론이 단독으로 얼마나 버티는지는 충분히 시험하지 못했다.
- **`auid>=1000` 필터 탓에 root로 직접 친 명령은 누락**될 수 있었다. 데몬 노이즈를 걷어내려고 의식적으로 받아들인 트레이드오프지만, 명령 감사의 사각지대인 건 분명하다.

## ④ 향후 시도

- **트래픽 레벨 가용성 방어** — nginx rate limiting·동시 연결 수 제한, WAF 또는 CloudFront/AWS Shield로 볼륨형·L7 공격을 완화한다. 공개 표면(노출 포트·서비스)을 최소화하고, 실제 운영 환경이라면 외부 스캔 노출도 줄인다.
- **IaC 완성 (도구별 역할 분리)** — 구성 자동화는 ansible 멱등 배포로 적용했고(nginx·방어 스크립트·systemd·auditd), 인프라 프로비저닝(NLB·EC2·타겟그룹)과 인스턴스 교체형(ASG·Launch Template)·Multi-AZ는 **Terraform**으로 코드화한다.
- **명령 감사 사각지대 보강** — `PROMPT_COMMAND` 같은 셸 레벨 훅은 root가 `unset` 한 줄로 우회할 수 있다. 커널에서 syscall을 추적하는 **eBPF 기반 런타임 보안**(Falco·Tetragon)이나 auditd 규칙 세분화(`auid=0` 직접 로그인 포함)로 root 직접 명령까지 잡는다.
- **클론 복구 강화** — `#2`·`#3`의 단독 복구를 별도 검증하고, `apt-mark hold` 등으로 패키지 선제 차단을 더한다.

## 의사결정 회고 — 가장 어려웠던 트레이드오프

방어를 "기능 추가"가 아니라 **선택과 대가**로 봤다. 경기에서 가장 고민했던 세 결정이다.

**1. 페일-오픈 vs 페일-클로즈 — "버틴다" 대신 "빠진다".**

헬스체크 `ok` 파일을 `chattr +i`로 잠그면 더 견고해 보인다. 하지만 잠그는 순간, 오염된 페이지가 200으로 채점봇에 그대로 노출된다.

그래서 일부러 잠그지 않았다. 오염되면 `ok`를 지워 503을 띄우고, NLB가 정상 노드로 우회하게 했다.

대가는 플래핑이다. 일시적 `curl` 타임아웃에도 `#1`이 격리될 수 있다(디바운스 미구현). "더 안전해 보이는 잠금"이 오히려 자폭이 될 수 있다는 걸, 실패 모드를 먼저 그려보고서야 알았다.

**2. 복구를 얼마나 공격적으로 — 되살리되 태우지는 않게.**

`kill` 연타에는 빠르게 되살려야 한다. 하지만 너무 공격적이면(주기 1초·무제한 재기동) 깨진 바이너리가 크래시 루프로 CPU를 태운다. 방어가 가용성을 깎는 셈이다.

그래서 systemd는 60초당 20회까지만 재기동하고(`[Unit] StartLimit`), 한도를 넘기면 watchdog(5초)와 cron(1분)이 `reset-failed` 후 이어받게 했다. watchdog도 주기 5초에 `curl` 1회로 가볍게 잡았다.

빠른 감지와 자원 안정성을 맞바꾼 선택이다.

**3. 로컬 방어를 "최종 방어선"이라 부르지 않기로 한 것.**

가장 중요한 결정은 코드가 아니라 **목표 설정**이었다. 16개 스크립트로 상호 부활 메시를 짜면서도, root 공격자가 원자적 스크립트 한 번에 이걸 다 지울 수 있다는 걸 알고 있었다.

그래서 로컬 계층의 목표를 "공격자 차단"이 아니라 **복구 시간 단축 + 오염 시 자가격리 트리거**로 낮춰 잡았다. 진짜 가용성은 `#2`/`#3` 클론과 NLB 페일오버에 맡겼다.

"단일 노드를 끝까지 살린다"는 환상 대신 "오염되면 빠지고 우회한다"를 택했다. 다중화의 대가(운영 복잡도·중복 재기동)를 알면서도 내린 이 판단이 프로젝트의 핵심이다.
